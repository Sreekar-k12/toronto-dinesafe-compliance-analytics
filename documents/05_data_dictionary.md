## Purpose
This Data Dictionary documents every field in the DineSafe dataset used 
in this project. It serves as the reference document for all analysts, 
developers, and stakeholders working with the data.

A data dictionary answers three questions for every field:
- **What is it?** — the definition
- **What does it look like?** — the format and example values
- **How is it used?** — the analytical purpose

## Historical Dataset Notes
The historical files (dinesafe_hist_2001.csv through dinesafe_hist_2022.csv) 
use a similar but not identical schema to the current dataset.

Key differences to handle during data integration:

| Issue | Description | Resolution |
|---|---|---|
| ID field | Historical files use oldEstId, current uses estId | Join on oldEstId when linking historical to current |
| Column names | May differ in capitalisation or naming | Standardise all column names to lowercase during loading |
| Severity codes | May use different encoding in older files | Map all severity values to standard M/S/C encoding |
| Date format | May vary across years | Parse all dates with pd.to_datetime(errors='coerce') |
| Missing fields | Some columns may not exist in older files | Add missing columns with null values during concat |

## Data Limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| No cuisine type field | KPI-07 uses violation category instead | Documented in KPI Framework |
| Dataset retired from portal | May have gaps in recent records | Use both current CSV and historical archive |
| Inconsistent naming in typeDesc | Same violation appears in multiple formats | Standardise during cleaning |
| Null coordinates for some records | Excluded from map visual | Document exclusion count in validation report |
| Historical schema differences | Joining across years may introduce errors | Document all join assumptions |
